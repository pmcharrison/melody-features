# Feature name and implementation audit

This document records candidate issues found during a read-only audit of the
exported feature table generated by `docs/quarto_table_build.py::build_table()`.
The goal is to catch features where the implementation, function name, display
name, docstring, or sign convention may mislead users.

These findings are intentionally conservative: they are not all confirmed bugs,
but each item appears worth triage before relying on the feature name as a
description of the returned value.

## Review method

- The feature table currently exports 246 rows.
- Read-only subagents reviewed bounded slices by category: pitch, interval,
  contour, duration/timing, duration/rhythm-pulse, tonality, complexity, and
  corpus.
- Each subagent compared feature names, display names, docstrings, source code,
  helper implementations, and available tests.
- This report records only questionable cases, not features that looked correct.

## Severity guide

- **High**: likely incorrect computation, inverted sign/convention, crash-prone
  edge case, or a name that materially misrepresents the returned value.
- **Medium**: likely misleading naming, duplicate/alias ambiguity, inconsistent
  code paths, or behavior that needs documentation before users can interpret it.
- **Low**: minor naming/documentation ambiguity, likely harmless convention
  mismatch, or lower-confidence parity concern.

## Suggested next steps

1. Triage high-severity items into confirmed bug, intended behavior needing docs,
   or false positive.
2. Add focused tests before changing behavior. Good first tests are:
   - polyrhythms complement/sign convention,
   - tonalness histogram over 24 correlations,
   - melodic and pitch histogram return shapes,
   - corpus max-order inclusion,
   - onset autocorrelation tempo sensitivity.
3. Decide an alias policy for duplicate rows. A small table-builder filter could
   remove duplicate function objects while preserving documented aliases.
4. Consider adding explicit display-name metadata rather than deriving all public
   names from `snake_case.title()`. That would address opaque names such as
   `complebm`, typo-derived names, and unit-bearing duration names.
